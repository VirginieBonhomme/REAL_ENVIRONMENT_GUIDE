NETWORK DESIGN CONCEPTS EXPLAINED

Every term and design decision in this lab explained from scratch. Read this section before touching a VM.

WHAT IS AN IP ADDRESS?
An IP address is a unique numerical label assigned to every device on a network, like a postal address for data. Your lab uses 192.168.10.x addresses — a private address range that only exists inside your lab. The internet has no idea these addresses exist and cannot route traffic to them directly. This is by design and is a foundational security property of your lab.
WHAT IS A SUBNET (192.168.10.0/24)?
A subnet defines a group of IP addresses that belong to the same network. The /24 means the first 24 bits are fixed (192.168.10) and the last 8 bits are variable (.0 through .255), giving you 254 usable addresses. Every VM in your lab sits in this subnet, which means they can talk directly to each other without going through a router. Think of /24 as "everyone in the same building."
WHAT IS A NAT NETWORK (VIRTUALBOX)?
NAT stands for Network Address Translation. In VirtualBox, a NAT Network allows all your VMs to share a single internet connection through the host machine (your HP Envy) AND communicate with each other. Your HP Envy acts as the router — it translates traffic between the VM's private 192.168.10.x addresses and your real internet IP. Security implication: VMs can reach the internet for downloads, and Kali can attack your Windows VMs — this is intentional for a red/blue lab.
NAT NETWORK vs HOST-ONLY vs BRIDGED
NAT Network: VMs reach internet + each other. Your choice — correct for this lab because Kali needs to attack other VMs. Host-Only: VMs talk to each other and your laptop only — no internet. Better isolation but Kali cannot pull tools. Bridged: VM appears directly on your home Wi-Fi — NEVER USE THIS. It exposes your vulnerable VMs to every device on your home network and potentially the internet.
WHAT IS DHCP?
DHCP stands for Dynamic Host Configuration Protocol. It automatically gives a device its network settings when it joins the network, instead of you typing them in manually. DHCP usually assigns the IP address, subnet mask, default gateway, and DNS server. In this lab, Target-PC can use DHCP because it is just a workstation target. Splunk and Kali use static IPs because your searches and detections need to find them at the same address every time.
WHAT IS A DOMAIN CONTROLLER?
A Domain Controller (DC) is a Windows Server that manages authentication and policy for an entire network domain. When you log into a domain-joined machine, the DC verifies your credentials. It runs Active Directory Domain Services (AD DS) — the database of every user, computer, and policy in the domain. Your DC is ADDC01 at 192.168.10.7. If an attacker compromises the DC, they own every account in the domain.
WHAT IS DNS AND WHY DOES AD DEPEND ON IT?
DNS (Domain Name System) translates names to IP addresses. Active Directory is entirely DNS-dependent — it uses DNS to locate the domain controller, find Kerberos authentication services, and resolve domain names. Your DC runs its own DNS server at 192.168.10.7. When you set a workstation's DNS to 192.168.10.7, it can resolve "redblue.local" and join the domain. Without correct DNS, domain joins fail with "domain not found" — the #1 troubleshooting issue in AD labs.
WHAT IS KERBEROS AUTHENTICATION?
Kerberos is the authentication protocol used by Active Directory. When you log in, the DC issues a Ticket-Granting Ticket (TGT) — a cryptographic token proving your identity. You then use the TGT to request service tickets for specific resources without re-entering your password. Why attackers target it: Stolen TGTs can impersonate users (Pass-the-Ticket). Compromised krbtgt account credentials allow forged tickets valid for any user (Golden Ticket attack).
WHAT IS LSASS.EXE AND WHY IS IT A TARGET?
LSASS (Local Security Authority Subsystem Service) is the Windows process responsible for enforcing security policy and managing credential storage. It holds password hashes and Kerberos tickets in memory. Tools like Mimikatz attack lsass.exe to dump these credentials — giving an attacker domain-wide access. This is why Sysmon EventCode 10 (Process Access targeting lsass.exe) is one of the highest-priority alerts in any SOC.
WHAT IS A SIEM?
SIEM stands for Security Information and Event Management. Splunk is your SIEM. It collects logs from every machine in your lab (via the Universal Forwarder), indexes them, and lets you search, correlate, and alert on suspicious patterns. The SIEM is the defender's central nervous system — every attack your Kali VM runs leaves traces on your endpoints, and those traces flow into Splunk where you can find and investigate them.
WHAT IS THE SPLUNK UNIVERSAL FORWARDER?
The Universal Forwarder is a lightweight Windows/Linux agent installed on each endpoint. It monitors configured log sources (Windows Event Logs, Sysmon logs) and ships them in real time to the Splunk indexer at 192.168.10.10 on port 9997. It uses very little RAM (under 50 MB) and CPU. Security aspect: The forwarder only sends data outbound — it cannot receive commands or execute code, making it a low-risk component on sensitive systems.
WHAT IS ATOMIC RED TEAM?
Atomic Red Team is an open-source library of small, safe, isolated attack simulations maintained by Red Canary. Each "atomic test" simulates one specific MITRE ATT&CK technique on your own system. You run them on your Windows VM to generate realistic attack telemetry, then verify your Splunk detections catch it. Security aspect: Tests are designed to be reversible — each has a Cleanup command that restores the system state after testing.
WHAT IS MITRE ATT&CK?
MITRE ATT&CK is a globally recognized framework cataloging real attacker tactics and techniques based on observed incidents. Every technique has a T-number (e.g., T1059.001 = PowerShell execution). It organizes techniques by tactic: Reconnaissance → Initial Access → Execution → Persistence → Privilege Escalation → Defense Evasion → Credential Access → Discovery → Lateral Movement → Collection → Exfiltration → Impact. In your lab, every Kali attack maps to one or more ATT&CK techniques.

SYSMON FULLY EXPLAINED

Sysmon is the most important tool in your detection stack. Understanding exactly what it captures and why is essential before you build any detection rules.

WHAT SYSMON IS — AND WHAT IT IS NOT
What it is: Sysmon (System Monitor) is a Windows system service and kernel-mode driver from Microsoft's Sysinternals suite — the same team that makes Process Explorer and Autoruns. It installs on a Windows machine and runs permanently in the background, even across reboots, writing detailed security events to a dedicated Windows Event Log channel: Applications and Services Logs → Microsoft → Windows → Sysmon → Operational.

What it is not: Sysmon does not analyze events, generate alerts, or block anything. It is purely a telemetry generator. It produces the raw data — your Splunk SIEM interprets it. This is an important conceptual distinction: Sysmon sees everything and records it; Splunk finds the patterns that indicate threats.

Why it matters over standard Windows logs: Standard Windows Security logs tell you "a process was created." Sysmon tells you which process, what arguments it was given, what its parent process was, the hash of the executable, and what network connection it made immediately afterward. That chain of information is what lets you reconstruct exactly what an attacker did.
Simple Difference — Wireshark vs Sysmon vs Splunk

Think of the same incident from three camera angles. Wireshark is the network camera. It shows the packets moving across the wire: IP addresses, ports, DNS, TCP flags, HTTP requests, TLS handshakes, and timing. Sysmon is the endpoint camera. It shows what happened inside a Windows machine: which process ran, what command line it used, what process launched it, what network connection it made, what registry key changed, and whether something touched lsass.exe. Splunk is the search room. It collects logs from many machines, lets you search them, correlate them, build timelines, and create detections.

TOOL WHAT IT ANSWERS PLAIN-LANGUAGE EXAMPLE
Wireshark What crossed the network? "I can see Kali sent SYN packets to this Windows host. I can see which ports replied with SYN-ACK. I can see an HTTP login sent a username and password in cleartext."
Sysmon What did this Windows endpoint do? "I can see powershell.exe ran on this host. I can see the full command line, the parent process, the user, and whether that process made a network connection."
Splunk What happened across all my logs? "I can search all Sysmon and Windows logs in one place, build a timeline, count failed logons, find which host was scanned, and write an alert for the pattern."
Interview Answer You Can Say Out Loud

"Wireshark, Sysmon, and Splunk do not show the same thing. Wireshark shows packet-level network evidence. Sysmon shows detailed endpoint activity from a Windows machine. Splunk is where I collect and search those logs across the lab. So if Kali runs a scan, Wireshark can show the actual SYN and SYN-ACK packets, Sysmon can show endpoint network events or related process activity, and Splunk lets me search and correlate the evidence into a timeline."

EID 1 Process Creation Every time a new process starts: full command line, executable hash, parent process name and ID, user who ran it, working directory, and session ID. The single most valuable Sysmon event. Catches PowerShell execution with encoded commands, cmd.exe spawning unusual children, and tools like Mimikatz or Nmap launching. Attackers modify file timestamps to make malware appear as if it was on the system for years, evading "new files" detection rules. Any legitimate process rarely changes file timestamps. T1070.006 Timestomping
EID 3 Network Connection Every outbound TCP/UDP connection: source and destination IP and port, the process that made the connection, and whether DNS was used. Catches Command and Control (C2) communication — malware calling home to an attacker server. Also reveals port scans: when Kali runs Nmap against your Windows box, hundreds of EID 3 events fire from the target's perspective showing all the connection attempts. Correlate with EID 1 to find which process made suspicious connections. T1046 Network ScanningT1071 C2 Communication
EID 4 Sysmon Service State Change When the Sysmon service starts or stops. An attacker who gains admin access may try to stop Sysmon to blind your detection. An unexpected EID 4 stop event — especially one not preceded by a maintenance window — is a high-priority alert. Also tells you if Sysmon crashed. T1562 Impair Defenses
EID 5 Process Terminated When a process exits: timestamp, process ID, and GUID. Combined with EID 1, gives you the full lifecycle of a process — when it started and when it ended. Useful for determining how long a suspicious process ran. Very short-lived processes (started and stopped in seconds) are common in attack chains where a tool executes and exits immediately. T1059 Short-lived execution
EID 6 Driver Loaded When a kernel driver loads: driver file path, hash, and whether it is signed by a trusted vendor. Kernel drivers run with the highest possible privileges. Legitimate systems load very few drivers, and they are almost always signed by Microsoft or major hardware vendors. An unsigned driver loading is extremely suspicious — rootkits and advanced malware use kernel drivers to hide from security tools. EID 6 events are rare, making every one worth investigating. T1543 Create/Modify System Process
EID 7 Image Loaded (DLL Load) When a DLL or executable image loads into a process: the DLL path, hash, and signature status. Disabled by default — generates very high volume. Used to detect DLL hijacking and side-loading — attack techniques where malicious code is placed in a DLL that a legitimate application loads. Also catches certain process injection techniques. Because this event is very noisy, it should only be enabled with a tight configuration file that filters to specific high-risk DLLs or processes. T1574 Hijack Execution Flow
EID 10 Process Access When one process opens another process — including what access rights were requested (read memory, write, etc.). The primary detection for Mimikatz and credential dumping. When Mimikatz or a similar tool tries to read lsass.exe memory to steal credential hashes, it must first open lsass.exe with ReadProcessMemory access — which generates EID 10 with TargetImage=lsass.exe. This is one of the most critical alerts in any SOC. Note: EID 10 can be noisy from legitimate diagnostic tools — filter to only flag lsass.exe as the target. T1003 OS Credential Dumping
EID 11 File Created When a file is created or overwritten: file path, process that created it, and hash of the file. Catches malware dropping payloads. Attackers often write tools to disk before executing them — a .exe or .ps1 file appearing in a temp directory or user profile is suspicious. Also used to monitor autostart locations like the Startup folder. Configure Sysmon to watch high-risk directories: C:\Users\*\Downloads, C:\Temp, C:\Windows\Temp, C:\Users\*\AppData\Roaming. T1105 Ingress Tool Transfer
EID 12 & 13 Registry Events EID 12: Registry key created or deleted. EID 13: Registry value set. Both capture the key path, value name, and the process making the change. Registry persistence is one of the most common attacker techniques. Malware adds entries to Run keys (HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Run) so it restarts after reboot. EID 12/13 catches these modifications in real time. Atomic Red Team's T1547.001 test specifically exercises this technique. T1547.001 Registry Run Keys
EID 15 File Stream Created When a file is created with an alternate data stream (ADS), specifically the Zone.Identifier stream added by browsers to downloaded files — called the Mark of the Web. Files downloaded from the internet carry a Zone.Identifier ADS marking them as potentially unsafe. Attackers who download tools via a browser generate this event. Also used to detect files being smuggled using ADS to hide content in legitimate files (T1564.004 NTFS ADS). T1564.004 NTFS ADS
EID 16 Sysmon Config Changed When the Sysmon configuration file is updated — who changed it and when. Provides auditability over your own detection setup. If an attacker modifies the Sysmon config to disable logging of their techniques, this event fires. It is also useful for tracking your own config changes during lab development. T1562 Impair Defenses
EID 22 DNS Query Every DNS query made by any process: the queried domain name, the process that made the query, and whether it succeeded. Catches C2 domain lookups — malware must resolve the attacker's domain before connecting. Also detects DNS tunneling (data exfiltration encoded in DNS queries) and beaconing behavior (regular periodic DNS queries to the same unusual domain). In your lab, when Kali runs Hydra or Metasploit, any DNS resolution for attacker infrastructure shows up here. T1071.004 DNS C2T1048 Exfil via DNS

SECURITY RULES

ABSOLUTE RULE — KALI SCOPE
Every Kali attack tool — Nmap, Hydra, Metasploit, CrackMapExec — must only target 192.168.10.0/24 addresses. Before every Kali session, run ip a and verify your IP is 192.168.10.250. Running these tools against any external IP is illegal under the Computer Fraud and Abuse Act regardless of intent. Document every attack with the exact command and timestamp.
NEVER USE BRIDGED ADAPTER
A Bridged Adapter makes your VM appear directly on your home Wi-Fi network — reachable from every phone, smart TV, and other device in your home. If your Windows VM has vulnerabilities (it does — that's the point), a Bridged adapter exposes them to everything on your network. All VMs use NAT Network only. If you ever see a Bridged adapter in VirtualBox settings for a lab VM, change it immediately.
SNAPSHOTS ARE MANDATORY
Take a VirtualBox snapshot before every step that changes system configuration. Right-click VM → Snapshots → Take. Name it descriptively: "Before-Sysmon-Install" or "After-GPO-Config". When something breaks — and it will — you revert to the last snapshot in 30 seconds instead of rebuilding from scratch for 3 hours. This is how professional lab environments are managed.
FIREWALL HARDENING ON SPLUNK VM
Ubuntu ships with UFW firewall installed but disabled. After Splunk installs, enable UFW with deny-all-incoming as the default, then allow only ports 22 (SSH), 8000 (Splunk Web UI), and 9997 (Forwarder) — all restricted to 192.168.10.0/24. This means even if something goes wrong with NAT isolation, Splunk ports are only reachable from inside the lab subnet.

How to Read Splunk SIEM Data

Read Splunk like an investigation timeline, not like a random list of logs. Start with the alert or question, lock the time window, identify the affected host and user, then pivot through related events before and after the suspicious activity. A single event rarely proves the whole incident. The analyst work is connecting events into a sequence: login, process execution, network connection, file or registry change, privilege use, and cleanup.

Search index=sysmon EventCode=1 to read process creation. Focus on Image, CommandLine, ParentImage, User, and ComputerName. Suspicious example: powershell.exe -EncodedCommand launched by Word or a browser. That suggests script execution from a phishing or download path.
EXAMPLE: PORT SCAN DETECTION
Search index=sysmon EventCode=3 and group by SourceIp, DestinationIp, and distinct DestinationPort. If one source IP touches dozens or hundreds of ports on one target in a short window, treat it as reconnaissance. In this lab, 192.168.10.250 should immediately stand out as Kali.
EXAMPLE: PASSWORD SPRAY
Search Windows failed logons with index=wineventlog EventCode=4625. A brute force is many failures against one account. A password spray is failures across many unique accounts from one source IP. Use dc(Account_Name) to count unique accounts.
EXAMPLE: LSASS ACCESS
Search index=sysmon EventCode=10 TargetImage="*lsass.exe". Focus on SourceImage, GrantedAccess, User, and ComputerName. Unknown tools reading lsass.exe on ADDC01 are critical because this may indicate credential dumping.

WHAT IS AN ORGANIZATIONAL UNIT (OU)?
An Organizational Unit is a container in Active Directory that holds user accounts, computer accounts, and other OUs. OUs mirror your organization's structure — IT, HR, Finance, SOC. The critical function of OUs is that Group Policy Objects (GPOs) link to OUs — meaning you can apply different security policies to different departments. An IT-Admins restriction GPO linked to OU=IT only affects IT users, not HR or Finance.
WHAT IS ROLE-BASED ACCESS CONTROL (RBAC)?
RBAC means permissions are assigned to security groups (roles), not to individual users. When a user changes roles, you modify their group membership — not every individual resource they access. This is how enterprise environments are managed. Least privilege means every account gets only the group memberships required for their current role — nothing more. Over-privileged accounts are one of the top security audit findings and one of the primary ways attackers escalate privileges after initial access.
WHAT IS A GROUP POLICY OBJECT (GPO)?
A GPO is a collection of policy settings that Windows applies automatically to computers and users in an OU. They enforce security configuration at scale across every machine in the domain. Examples: account lockout policy (locks accounts after N failed logins — prevents brute-force), password complexity requirements, disabling Control Panel for non-admin users, and deploying software. GPOs are applied at login and refreshed every 90 minutes. Security importance: GPOs are how you enforce consistent security across hundreds of endpoints without touching each one individually.
WHAT IS THE DISTINGUISHED NAME (DN)?
The Distinguished Name is Active Directory's full address format for any object. It reads right to left from the domain root: CN=alice.johnson,OU=IT,OU=CORP,DC=redblue,DC=local. DC=redblue,DC=local is the domain root. OU=CORP,OU=IT are the containing OUs. CN=alice.johnson is the specific object. You use DNs in PowerShell commands to specify exactly where to create or move objects. Getting the DN wrong is a common PowerShell error — always verify with Get-ADUser first.

STEP 1 — UNDERSTAND THE CONCEPT BEFORE WRITING THE SCRIPT
WHAT IS A CSV FILE?
CSV stands for Comma-Separated Values. It is a plain text file where each row is a record and each column is separated by a comma. Excel can open them. PowerShell can read them with Import-Csv. In IAM, HR systems export new hire data as CSV files that automation scripts consume. Example row: marcus.webb,Marcus,Webb,Finance,carol.white
WHAT IS A FOREACH LOOP?
A foreach loop runs the same block of code once for every item in a list. foreach ($user in $userList) means: take the first user, run all the code inside the curly braces, then take the second user, run all the code again, and repeat until the list is empty. This is how you create 50 accounts with one script run instead of 50 manual steps.
WHAT IS ConvertTo-SecureString?
PowerShell cannot pass a plain text password directly to New-ADUser for security reasons. ConvertTo-SecureString wraps the plain text password in a secure object that PowerShell's security cmdlets accept. The -AsPlainText -Force flags tell PowerShell you understand the password is currently in plain text and you accept the responsibility.
WHAT IS -ChangePasswordAtLogon?
Setting -ChangePasswordAtLogon $true forces the user to set a new password on their very first login. This means the IT team never knows the user's actual password — only the temporary one used to provision the account. This is a security best practice and is required for compliance in most audit frameworks.

MOVER PROCESS — IAM LIFECYCLE
Role Change Script — Department Transfer Automation
25 MIN
+

When an employee changes departments, their AD account must be moved to the new OU, old group memberships removed, and new group memberships added. This is the Mover step of the JML lifecycle. Doing this manually risks missing a group and leaving access creep — which is a compliance finding. A script ensures every step happens consistently every time.

WHAT IS ACCESS CREEP?
Access creep is when a user accumulates permissions from previous roles that were never removed. An employee who moved from Finance to IT three years ago may still have Finance share access. Over time this creates significant over-privilege. Access creep is one of the top IAM audit findings and is why Mover processes must remove old access before adding new access.
WHAT IS Move-ADObject?
Move-ADObject changes which OU a user or computer object lives in without recreating the account. The user keeps their SID (Security Identifier — the internal unique ID Windows actually uses for permissions), their group history, and their password. Only their location in the directory tree changes. This is how you move a user from OU=Finance to OU=IT without losing any of their account settings.

Save as C:\\Scripts\\Move-UserDepartment.ps1

# ============================================================
# SCRIPT: Move-UserDepartment.ps1
# PURPOSE: Move user between departments — JML Mover Process
# USAGE: .\\Move-UserDepartment.ps1 -Username "sarah.jones" -NewDept "IT" -OldGroup "Finance-Users" -NewGroup "IT-Admins"
# ============================================================
 
# PARAMETERS — these are passed in when you call the script
param(
    [Parameter(Mandatory=$true)]  [string]$Username,
    [Parameter(Mandatory=$true)]  [string]$NewDepartment,
    [Parameter(Mandatory=$true)]  [string]$OldGroupName,
    [Parameter(Mandatory=$true)]  [string]$NewGroupName,
    [Parameter(Mandatory=$false)] [string]$NewManager = $null
)
# Mandatory=$true means the script errors if you forget to pass that value
# Mandatory=$false means it is optional
 
$DomainBase = "DC=redblue,DC=local"
$LogFile = "C:\\AuditReports\\Mover-$Username-$(Get-Date -Format 'yyyyMMdd-HHmm').log"
 
# Log the request
"=== Department Transfer Log ===" | Out-File $LogFile
"Date: $(Get-Date)" | Out-File $LogFile -Append
"User: $Username" | Out-File $LogFile -Append
"New Department: $NewDepartment" | Out-File $LogFile -Append
"" | Out-File $LogFile -Append
 
# Get the user object — store it for use throughout the script
$User = Get-ADUser -Identity $Username -Properties MemberOf, Department, DistinguishedName
"PRE-CHANGE STATE:" | Out-File $LogFile -Append
"  Current Department: $($User.Department)" | Out-File $LogFile -Append
"  Current Groups: $($User.MemberOf -join ', ')" | Out-File $LogFile -Append
 
# STEP 1 — Remove from old group FIRST (prevent access creep)
try {
    Remove-ADGroupMember -Identity $OldGroupName -Members $Username -Confirm:$false
    $msg = "[STEP 1 OK] Removed from group: $OldGroupName"
    Write-Host $msg -ForegroundColor Green
    $msg | Out-File $LogFile -Append
} catch { "[STEP 1 FAILED] $($_.Exception.Message)" | Out-File $LogFile -Append }
 
# STEP 2 — Add to new group
try {
    Add-ADGroupMember -Identity $NewGroupName -Members $Username
    $msg = "[STEP 2 OK] Added to group: $NewGroupName"
    Write-Host $msg -ForegroundColor Green
    $msg | Out-File $LogFile -Append
} catch { "[STEP 2 FAILED] $($_.Exception.Message)" | Out-File $LogFile -Append }
 
# STEP 3 — Update Department attribute
try {
    Set-ADUser -Identity $Username -Department $NewDepartment
    $msg = "[STEP 3 OK] Department updated to: $NewDepartment"
    Write-Host $msg -ForegroundColor Green
    $msg | Out-File $LogFile -Append
} catch { "[STEP 3 FAILED] $($_.Exception.Message)" | Out-File $LogFile -Append }
 
# STEP 4 — Move to new OU
try {
    $NewOUPath = "OU=$NewDepartment,OU=CORP,$DomainBase"
    Move-ADObject -Identity $User.DistinguishedName -TargetPath $NewOUPath
    $msg = "[STEP 4 OK] Moved to OU: $NewOUPath"
    Write-Host $msg -ForegroundColor Green
    $msg | Out-File $LogFile -Append
} catch { "[STEP 4 FAILED] $($_.Exception.Message)" | Out-File $LogFile -Append }
 
# STEP 5 — Update Manager if provided
if ($NewManager) {
    try {
        $ManagerDN = (Get-ADUser $NewManager).DistinguishedName
        Set-ADUser -Identity $Username -Manager $ManagerDN
        $msg = "[STEP 5 OK] Manager updated to: $NewManager"
        Write-Host $msg -ForegroundColor Green
        $msg | Out-File $LogFile -Append
    } catch { "[STEP 5 FAILED] $($_.Exception.Message)" | Out-File $LogFile -Append }
}
 
# Log the post-change state for audit trail
$Updated = Get-ADUser -Identity $Username -Properties MemberOf, Department, DistinguishedName
"" | Out-File $LogFile -Append
"POST-CHANGE STATE:" | Out-File $LogFile -Append
"  New Department: $($Updated.Department)" | Out-File $LogFile -Append
"  New Location: $($Updated.DistinguishedName)" | Out-File $LogFile -Append
"  Current Groups: $($Updated.MemberOf -join ', ')" | Out-File $LogFile -Append
"=== Transfer complete $(Get-Date) ===" | Out-File $LogFile -Append
Write-Host "Log: $LogFile" -ForegroundColor Cyan

# USAGE: .\\Invoke-Offboarding.ps1 -Username "tom.baker" -Reason "Voluntary resignation"
# ============================================================

)

"=============================================" | Out-File $LogFile -Append

"Requested By: $RequestedBy" | Out-File $LogFile -Append
"Start Time:   $StartTime" | Out-File $LogFile -Append
"" | Out-File $LogFile -Append

$User = Get-ADUser -Identity $Username -Properties MemberOf, Department, DistinguishedName, Manager
"PRE-OFFBOARDING STATE:" | Out-File $LogFile -Append
"  Enabled: $($User.Enabled)" | Out-File $LogFile -Append
"  Department: $($User.Department)" | Out-File $LogFile -Append
"  DN: $($User.DistinguishedName)" | Out-File $LogFile -Append

"" | Out-File $LogFile -Append

Disable-ADAccount -Identity $Username
$t = Get-Date -Format "HH:mm:ss"
$msg = "[$t STEP 1] Account DISABLED"
Write-Host $msg -ForegroundColor Yellow
$msg | Out-File $LogFile -Append

$msg = "[$t STEP 2] Password RESET to random string (not recorded)"
Write-Host $msg -ForegroundColor Yellow
$msg | Out-File $LogFile -Append

    Remove-ADGroupMember -Identity $Group -Members $Username -Confirm:$false
    $RemovedGroups += $GroupName

ACCESS GOVERNANCE — IAM CORE FUNCTION
Stale Account Detection and Access Review Report
20 MIN
+

Access reviews are a core IAM function. This script identifies accounts that have not been used in 90 days, accounts with PasswordNeverExpires enabled, and accounts with excessive group memberships. These are the three most common findings in IAM audits. Run this script quarterly and document the output — it is your access governance evidence.

WHAT IS AN ACCESS REVIEW?
An access review (also called access certification) is a periodic process where managers or system owners certify that their users still need their current access. Automated reports identify candidates for review. The manager reviews each account and either certifies it (keep access) or revokes it (remove access). This is required by SOC 2, ISO 27001, and HIPAA frameworks.
WHAT IS LastLogonDate?
LastLogonDate is an AD attribute that records the last time a user authenticated against the domain. Accounts with LastLogonDate older than 90 days are considered stale — they may belong to contractors who left, employees on extended leave, or service accounts that are no longer needed. Stale accounts that remain active are an attack surface.

Save as C:\\Scripts\\Get-AccessReviewReport.ps1

# ============================================================
# SCRIPT: Get-AccessReviewReport.ps1
# PURPOSE: Quarterly access review — IAM governance reporting
# SCHEDULE: Run quarterly (see Script 6 for scheduling)
# ============================================================
 
$ReportDate = Get-Date -Format "yyyy-MM-dd"
$ReportPath = "C:\\AuditReports\\AccessReview-$ReportDate"
New-Item -ItemType Directory -Path $ReportPath -Force | Out-Null
 
Write-Host "=== REDBLUE ACCESS REVIEW REPORT ===" -ForegroundColor Cyan
Write-Host "Date: $ReportDate" -ForegroundColor Cyan
Write-Host "" 
 
# FINDING 1 — Stale accounts (no logon in 90 days)
Write-Host "FINDING 1: Accounts inactive 90+ days" -ForegroundColor Yellow
$StaleAccounts = Get-ADUser -Filter {Enabled -eq $true} -Properties LastLogonDate, Department |
    Where-Object { $_.LastLogonDate -lt (Get-Date).AddDays(-90) -or $_.LastLogonDate -eq $null } |
    Select-Object Name, SamAccountName, Department, LastLogonDate, DistinguishedName
 
$StaleAccounts | Format-Table -AutoSize
$StaleAccounts | Export-Csv "$ReportPath\\Stale-Accounts.csv" -NoTypeInformation
Write-Host "  Found: $($StaleAccounts.Count) stale accounts — saved to Stale-Accounts.csv" -ForegroundColor White
Write-Host ""
 
# FINDING 2 — PasswordNeverExpires (compliance risk)
Write-Host "FINDING 2: Accounts with PasswordNeverExpires enabled" -ForegroundColor Yellow
$NeverExpire = Get-ADUser -Filter {PasswordNeverExpires -eq $true} -Properties PasswordNeverExpires, Department, LastLogonDate |
    Select-Object Name, SamAccountName, Department, LastLogonDate
 
$NeverExpire | Format-Table -AutoSize
$NeverExpire | Export-Csv "$ReportPath\\PasswordNeverExpires.csv" -NoTypeInformation
Write-Host "  Found: $($NeverExpire.Count) accounts — saved to PasswordNeverExpires.csv" -ForegroundColor White
Write-Host ""
 
# FINDING 3 — Over-privileged accounts (member of 3+ groups beyond Domain Users)
Write-Host "FINDING 3: Potentially over-privileged accounts (3+ group memberships)" -ForegroundColor Yellow
$AllUsers = Get-ADUser -Filter * -Properties MemberOf, Department
$OverPriv = $AllUsers | Where-Object { $_.MemberOf.Count -ge 3 } |
    Select-Object Name, SamAccountName, Department,
        @{Name='GroupCount'; Expression={$_.MemberOf.Count}},
        @{Name='Groups'; Expression={($_.MemberOf | ForEach-Object { (Get-ADGroup $_).Name }) -join ' | '}}
 
$OverPriv | Format-Table Name, SamAccountName, GroupCount -AutoSize
$OverPriv | Export-Csv "$ReportPath\\OverPrivileged.csv" -NoTypeInformation
Write-Host "  Found: $($OverPriv.Count) accounts — saved to OverPrivileged.csv" -ForegroundColor White
Write-Host ""
 
# FINDING 4 — Domain Admins list (always review this)
Write-Host "FINDING 4: Current Domain Admins" -ForegroundColor Yellow
$DomAdmins = Get-ADGroupMember "Domain Admins" | Select-Object Name, SamAccountName, objectClass
$DomAdmins | Format-Table -AutoSize
$DomAdmins | Export-Csv "$ReportPath\\DomainAdmins.csv" -NoTypeInformation
Write-Host "  Found: $($DomAdmins.Count) Domain Admins — saved to DomainAdmins.csv" -ForegroundColor White
Write-Host ""
 
# SUMMARY
$Summary = @"
ACCESS REVIEW SUMMARY — $ReportDate
Domain: redblue.local
==============================
Stale accounts (90+ days):          $($StaleAccounts.Count)
PasswordNeverExpires accounts:       $($NeverExpire.Count)
Over-privileged accounts (3+ groups): $($OverPriv.Count)
Domain Admin count:                  $($DomAdmins.Count)
==============================
All reports saved to: $ReportPath
"@
Write-Host $Summary -ForegroundColor Cyan
$Summary | Out-File "$ReportPath\\Summary.txt"

# PURPOSE: Identify passwords expiring in the next 14 days

$DaysWarning = 14
$Today = Get-Date
$ReportFile = "C:\\AuditReports\\PasswordExpiry-$(Get-Date -Format 'yyyyMMdd').csv"

$MaxAge = (Get-ADDefaultDomainPasswordPolicy).MaxPasswordAge.Days

# Find users with expiring passwords
$ExpiringUsers = Get-ADUser -Filter {Enabled -eq $true -and PasswordNeverExpires -eq $false} `
    -Properties PasswordLastSet, Department, EmailAddress |
    Where-Object { $_.PasswordLastSet -ne $null } |
    ForEach-Object {
        $ExpiryDate = $_.PasswordLastSet.AddDays($MaxAge)
        $DaysLeft = ($ExpiryDate - $Today).Days

WHAT IS TASK SCHEDULER?
Task Scheduler is a Windows service that runs programs, scripts, or commands at scheduled times or in response to system events. Think of it as an alarm clock for your scripts. In enterprise environments it is used to run backup jobs, compliance reports, monitoring scripts, and IAM access reviews automatically. You can also schedule tasks via PowerShell — which creates a documented, repeatable setup process.
WHAT IS -RunLevel Highest?
-RunLevel Highest tells Task Scheduler to run the task with Administrator-level privileges. Since AD cmdlets require admin rights, your scheduled script must run elevated. Without this, the script runs with standard user permissions and every AD operation fails with an access denied error.

WHAT IS WIRESHARK?
Wireshark is a free, open-source network protocol analyzer. It captures raw packets from a network interface and lets you examine every field of every packet in detail. Unlike Splunk which shows you endpoint logs, Wireshark shows you what was actually on the wire — the exact bytes that traversed the network. Wireshark requires Npcap (Windows packet capture library) to intercept live traffic — the installer bundles it automatically.
TCP THREE-WAY HANDSHAKE
TCP establishes every connection with a three-packet handshake: SYN (client says "I want to connect") → SYN-ACK (server says "ready") → ACK (client says "connected"). A port scan sends SYN packets to many ports rapidly and watches for SYN-ACK responses (open port) or RST responses (closed port). In Wireshark, filter tcp.flags.syn==1 and tcp.flags.ack==0 to find initial SYNs — a flood of them from one IP to sequential ports is a port scan signature.
WHAT HTTPS HIDES AND WHAT IT DOES NOT
HTTPS encrypts: URL paths, HTTP headers, cookies, credentials, response bodies. HTTPS still reveals: destination hostname (via Server Name Indication — SNI — in the TLS ClientHello packet), destination IP address, traffic volume and timing. In Wireshark, filter tls.handshake.type==1 and look in the ClientHello for the SNI field — the destination domain is visible in plaintext even though the content is encrypted.
WHY DNS IS ALWAYS VISIBLE
DNS (Domain Name System) queries are sent in cleartext before the HTTPS connection is established. This means even though HTTPS encrypts the content, an observer can see every domain name you looked up — revealing which websites you visited. This is why DNS-over-HTTPS (DoH) was created — it encrypts the DNS lookup itself. In your lab, every tool Kali runs that needs a hostname generates a DNS query visible in Wireshark filter dns.
How to Read Network Traffic

Do not start by clicking random packets. Start wide, then narrow down. First identify the time window, top talkers, protocols, and unusual volume. Then filter to the suspicious conversation and read the packet sequence from top to bottom like a timeline: who initiated, who responded, what protocol was used, what data was exposed, and whether the behavior matches normal baseline traffic.

Packet Evidence vs Endpoint Evidence

Wireshark sees packet-level evidence: raw network frames, TCP flags, DNS queries, HTTP requests, TLS handshakes, timing, and byte-level conversations. Sysmon and Splunk see endpoint/log evidence: process creation, network connections recorded by the host, logon events, registry changes, and process access events. Use Wireshark to prove what crossed the network; use Splunk/Sysmon to prove which host, user, and process caused it. The strongest investigation uses both views together.

EXAMPLE: NMAP PORT SCAN
Look for one source IP sending many SYN packets to one target across many destination ports. Open ports respond with SYN-ACK; closed ports respond with RST. The analyst finding should name the attacker IP, target IP, port range, open ports discovered, and scan time window.
EXAMPLE: HTTP PASSWORD EXPOSURE
Filter for http.request.method=="POST", click the POST packet, and expand the form fields. If you can read a username or password, the risk is high because any on-path observer could also read it. Your finding should include the site, timestamp, exposed fields, and immediate action: change the password.
EXAMPLE: SUSPICIOUS DNS
Filter dns and review the queried names. Watch for repeated failed lookups, strange random-looking domains, domains unrelated to the user's work, or one host making many DNS requests in a short time. DNS is often the first clue before a web connection or malware callback.
EXAMPLE: HTTPS CONNECTION
For HTTPS, you usually cannot read page content or passwords. Instead, analyze metadata: destination IP, SNI hostname, certificate details, connection timing, and data volume. A large upload to an unusual hostname may be more important than the encrypted bytes themselves.

CREDENTIAL ATTACK — T1110.003
Password Spray Against redblue Domain
25 MIN
+

A password spray tries one common password against many accounts — staying below the lockout threshold per account to avoid detection. Run it against redblue domain users, then detect the pattern in Splunk. The detection logic is fundamentally different from brute-force detection.

# ATTACK — Kali: spray one password against multiple redblue accounts
# Create a small user list first:
echo -e "analyst.one\nanalyst.two\nalice.johnson\nbob.smith" > /tmp/userlist.txt
crackmapexec smb 192.168.10.7 -u /tmp/userlist.txt -p 'Password123' --continue-on-success
# MITRE ATT&CK: T1110.003 — Password Spraying
 
# DETECT — Splunk:
# Password SPRAY signature: many UNIQUE accounts, few attempts each, from one IP
# (vs brute-force: one account, many attempts)
index=wineventlog EventCode=4625
| bucket _time span=1m
| stats dc(Account_Name) as unique_accounts, count by _time, IpAddress
| where unique_accounts > 3
# If unique_accounts is high → password spray
# If count is high for ONE account → brute-force
# This distinction is critical for accurate detection
KEY DETECTION INSIGHT The dc() (distinct count) function is what makes spray detection work. Brute-force = high count, low dc(Account_Name). Spray = high dc(Account_Name), low count per account. Getting this wrong means you miss one of them.

LATERAL MOVEMENT — T1021.002
Remote Execution via SMB Using Valid Credentials
25 MIN
+

If credential attacks succeed, an attacker moves laterally — connecting to other systems using the obtained credentials. This generates Logon Type 3 (network logon) events in Windows Security logs and process creation events in Sysmon. The combination of both is the lateral movement signature.

# ATTACK — Kali: execute a command on Windows 10 using valid credentials:
crackmapexec smb 192.168.10.[windows-ip] -u analyst.one -p 'Lab@Pass2024!' -x "whoami /all"
# MITRE ATT&CK: T1021.002 — Remote Services: SMB/Windows Admin Shares
 
# DETECT — Splunk: remote logon from Kali IP
index=wineventlog EventCode=4624 LogonType=3 IpAddress="192.168.10.250"
# LogonType=3 = network logon (remote access from another machine)
 
# Find the executed command in Sysmon:
index=sysmon EventCode=1 ComputerName=[windows-hostname]
| table _time, Image, ParentImage, CommandLine, User | sort _time
 
# The full lateral movement chain in one query:
index=wineventlog EventCode=4624 LogonType=3 IpAddress="192.168.10.250"
| append [search index=sysmon EventCode=1 User="redblue\\analyst.one"]
| table _time, EventCode, User, IpAddress, Image, CommandLine | sort _time
# This shows: remote logon → process execution in one combined timeline
CORRELATION EventCode 4624 Logon Type 3 from Kali's IP immediately followed by Sysmon EventCode 1 process creation is the lateral movement signature. Document this correlation — it is what Tier 2 analysts look for in an IR investigation.

LAB 01 — SYSMON

"What is Sysmon and how is it different from standard Windows Event Logs?"

YOUR ANSWER

Sysmon is a Windows system service and kernel-mode driver from Microsoft's Sysinternals suite. It installs on a Windows endpoint and runs permanently in the background, writing detailed security events to a dedicated log channel — Applications and Services Logs, Microsoft, Windows, Sysmon, Operational. Standard Windows logs tell you "a process was created." Sysmon tells you which process, what arguments it was given, what its parent process was, the hash of the executable, and what network connection it made immediately after. That chain of detail is what lets you reconstruct exactly what an attacker did. In my lab, when Kali ran Nmap against my Windows machine, Sysmon EventCode 3 generated hundreds of network connection events I could analyze in Splunk. Standard Windows logs would have missed the scan entirely.

LAB 01 — DETECTION

"What is the difference between a brute-force attack and a password spray, and how do you detect each in Splunk?"

YOUR ANSWER

A brute-force attack tries many passwords against one account — you detect it by finding many EventCode 4625 events for a single Account_Name. A password spray tries one common password against many accounts to stay below the lockout threshold per account — you detect it by finding EventCode 4625 events spread across many unique Account_Names from a single source IP. In my lab I ran both: Hydra for brute-force and CrackMapExec for password spray. The Splunk detection is completely different: brute-force uses count where count exceeds threshold; spray uses dc(Account_Name) — distinct count of unique accounts — where that number exceeds threshold from one IP. Using the wrong detection logic means you miss one of them entirely.

LAB 02 — ACTIVE DIRECTORY

"Walk me through how you would handle a terminated employee who was involved in a security incident."

YOUR ANSWER

Speed is critical — every minute the account stays active is a risk window. First action is Disable-ADAccount immediately, then verify Enabled=False before touching anything else. LAB 04 — MITRE ATT&CK

"What is MITRE ATT&CK and how did you use it in your lab?"

YOUR ANSWER

MITRE ATT&CK is a knowledge base of adversary tactics and techniques cataloged from real-world observations. It organizes attacks by tactic — reconnaissance, initial access, execution, persistence, credential access, lateral movement — and assigns each technique a T-number. In my lab I used it as both an attack guide and a detection framework. When I ran Nmap from Kali, that's T1046 — Network Service Discovery. When I ran a password spray with CrackMapExec, that's T1110.003. For each attack I ran, I documented what Sysmon EventCode fired and whether my Splunk detection rules covered it. The result was a MITRE coverage matrix showing my detection capabilities and gaps. That's the same analysis security engineers produce when evaluating enterprise detection programs.
